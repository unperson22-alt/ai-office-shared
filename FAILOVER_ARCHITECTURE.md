## 0. Контекст и допущения

Офис уже работает по failover-логике, но она не описана как стандарт. Фактическое состояние
(из `SYSTEM_STATE.md` и `shared/ollama.py`):

- **Мозг ботов** — Anthropic Claude. Тяжёлые решения → `claude-opus-4-8`; рабочие вызовы →
  `claude-sonnet-4-6` / `claude-haiku-4-5-20251001`.
- **Локальный fallback** — Ollama `gemma3:4b` через `try_ollama()` (fail-silent: при любой
  ошибке возвращает `None` → бот идёт в Anthropic).
- **Ретраи** — уже применяются точечно (советники Тилли: 1 попытка + backoff 2 с).
- Хостинг — Railway; шина — Redis; роутер — Филли; супервизор/мониторинг — Силли.

**Принятые допущения (зафиксированы, т.к. данных о трафике нет):**

| # | Допущение | Значение по умолчанию |
|---|---|---|
| A1 | Профиль нагрузки | низкочастотные интерактивные запросы (боты Telegram), пики при деплоях |
| A2 | SLA на ответ | интерактив ≤ 30 с; фоновые отчёты ≤ несколько минут |
| A3 | Бюджет важнее латентности на «рабочих» вызовах; качество важнее бюджета на «финальных» |
| A4 | Триггеры failover | `Timeout`, `RateLimit (429)`, `5xx/overloaded (500/529)`, сетевые обрывы, `refusal` |
| A5 | Цены не-Anthropic моделей | ориентир, **подлежат проверке** (см. §5.1, knowledge cutoff) |

> ⚠️ **Актуальность ландшафта.** Список из ТЗ (o1/o3/4o, Claude 3.5, Gemini 1.5, DeepSeek V3/R1)
> отражает 2024–2025. На дату стандарта (2026-06) офис уже на поколении Claude 4.x
> (`opus-4-8`, `sonnet-4-6`, `haiku-4-5`). Ниже модели сведены в **единую лестницу мощности**:
> разобран список ТЗ и добавлен фактический парк офиса, чтобы failover-цепочки были рабочими, а не историческими.

## 2. Карта зависимостей и отказов

**Как выбор модели влияет на систему:**

| Ось | Мощная (opus/o3/R1) | Стандартная (haiku/flash/4o-mini) | Локальная (gemma3:4b) |
|---|---|---|---|
| Латентность | выше (reasoning) | низкая | низкая, но зависит от железа |
| Стоимость 1M | высокая | низкая | ~0 (свой сервер) |
| Лимиты (RPM/TPM) | жёстче, дороже поднимать | свободнее | нет внешних лимитов |
| Риск отказа | rate-limit на пиках, overload | реже | отказ железа/OOM |

**Карта отказов → реакция:**

| Симптом | Класс | Реакция (см. §5.4) |
|---|---|---|
| `Timeout` | transient | ретрай (backoff 2→4→8 с), затем downgrade |
| `429 RateLimit` | transient/capacity | уважать `retry-after`; параллельно — активный бэкап |
| `500/529 overloaded` | provider | сразу активный бэкап (не ждать) |
| сетевой обрыв | infra | ретрай → бэкап → Ollama |
| `refusal` (safety) | policy | не ретраить тем же промтом; opt-in серверный fallback или эскалация |
| бэкап тоже упал | каскад | **сквозной критический алерт владельцу** (§5.4.3) |

## 3. Критерии успеха

**Нормальный режим (GREEN):**
- Запрос обслужен моделью своего уровня (роль = модель по §5.2).
- `p95` латентности в рамках SLA (A2); стоимость в рамках бюджета роли.
- Реакции качества пишутся в `office:quality:*` как обычно.

**Деградация (YELLOW — Graceful Degradation):**
- Основная модель недоступна → ответ выдан бэкапом **с пометкой о пониженном уровне**.
- Требования к формату ослаблены контролируемо (§5.4.2), но контракт ответа не сломан.
- В лог пишется **WARNING** с указанием: роль, основная модель, фактическая модель, причина.
- Бот **не падает**, пользователь получает полезный (пусть более простой) ответ.

**Авария (RED — каскадный отказ):**
- Бэкап тоже не справился → пользователю отдаётся **явный критический алерт** (а не пустота/стектрейс).
- Владельцу (`YOUR_TELEGRAM_ID`) уходит сквозной алерт с диагностикой (§5.4.3).
- Запрос помечается как degraded в Redis для последующего разбора.

## 4. Делегирование команде (автономные ТЗ)

- **Филли (роутер):** при `5xx` от целевого бота — ретрай по тому же маршруту (backoff), затем
  деградация маршрута; в эфир/группу не флудить (передавать `source=CLAUDE`).
- **Силли (супервизор):** в `CILLY_MONITOR_PAUSED`-режиме не считать conflict-ошибки рестартов
  багами; различать «модель деградировала» (WARNING) и «каскад» (CRITICAL → эскалация владельцу).
- **Каждый бот:** вызывать LLM **только** через `call_llm()` (§5.4), а не напрямую `anthropic.messages.create`.
  При смене уровня модели — **переписывать промт под уровень** (§5.4.2), а не слать opus-промт в gemma.
- **Тилли/Пророк (деньги-adjacent):** финальные вердикты — `opus-4-8`; при недоступности **не**
  «молча терять» советников — честно сообщать модели о деградации и понижать уверенность вывода.

### 5.3 Максимальный функционал

**Строгие промты.** Промт пишется под **уровень** модели (см. §5.4.2). Для opus-4-8: давать полную
постановку в первом ходе, не дробить; избегать `CRITICAL: YOU MUST` (4.x перетриггеривает).

**Structured Outputs (JSON Mode).** Гарантированный JSON — через `output_config.format`
(не устаревший `output_format`). Схема: `type:object`, `additionalProperties:false`, `required`.

```python
resp = client.messages.create(
    model="claude-opus-4-8", max_tokens=4096,
    output_config={"format": {"type": "json_schema", "schema": VERDICT_SCHEMA}},
    messages=[{"role": "user", "content": prompt}],
)
```

**Function Calling / Tool Use.** Для строгой валидации параметров — `strict: True` **на самом
инструменте** (не на `tool_choice`), со схемой `additionalProperties:false` + `required`.
Параллельные результаты инструментов — **в одном** user-сообщении. Парсить `tool_use.input`
через `json.loads` (никакого raw-string match — экранирование различается).

> ⚠️ Префилл ассистента и `thinking budget_tokens` на 4.x → 400. Использовать
> `thinking:{type:"adaptive"}` + `output_config.effort`; формат — через structured outputs.

### 5.4 Архитектура сбоеустойчивости (FAILOVER LOGIC)

Единая точка входа `call_llm()` (расширение текущего `try_ollama → anthropic`). Никаких прямых
вызовов провайдера в боте.

#### 5.4.1 Матрица дублирования (кто кого заменяет)

Цепочка = [основная] → [активный бэкап того же класса] → [downgrade на класс ниже] → [Ollama] → [алерт].

| Роль / класс | Основная | Бэкап-1 (тот же класс) | Бэкап-2 (downgrade) | Последний рубеж |
|---|---|---|---|---|
| Стратег (S) | `claude-opus-4-8` | `claude-fable-5` *или* `o3`¹ | `claude-sonnet-4-6` | `gemma3:4b` → алерт |
| Кодер (S) | `claude-opus-4-8` | `claude-sonnet-4-6` | `deepseek-chat`¹ | `gemma3:4b` → алерт |
| Креатив/Данные (A) | `claude-sonnet-4-6` | `gpt-4o`¹ / `gemini-1.5-pro`¹ | `claude-haiku-4-5-20251001` | `gemma3:4b` → алерт |
| Оператор (C) | `claude-haiku-4-5-20251001` | `gemini-1.5-flash`¹ / `gpt-4o-mini`¹ | — | `gemma3:4b` → алерт |

> Бэкап «активный» = ключ провайдера сконфигурирован и протестирован, переключение мгновенное.
> Если у офиса нет ключей OpenAI/Google/DeepSeek — цепочка вырождается в `opus → sonnet → haiku → gemma`,
> что **уже** покрыто текущим парком. Не-Anthropic строки — точки расширения, включаются по мере добавления ключей.

Конфиг (декларативно, читается ботом из env/Redis):

```yaml
# failover_chains.yaml — источник истины для call_llm()
strategist:
  primary:  {provider: anthropic, model: claude-opus-4-8, effort: xhigh}
  backups:  [{provider: anthropic, model: claude-sonnet-4-6}]   # + o3/fable при наличии ключа
  local:    {provider: ollama, model: gemma3:4b}
operator:
  primary:  {provider: anthropic, model: claude-haiku-4-5-20251001}
  backups:  []
  local:    {provider: ollama, model: gemma3:4b}
retry: {attempts: 3, backoff_s: [2, 4, 8]}     # transient: timeout/429/5xx
escalate_to: YOUR_TELEGRAM_ID
```

#### 5.4.2 Алгоритм «Делай что можешь» (Graceful Degradation)

При переходе на модель **слабее** основной — адаптировать промт и **снизить требования к результату**,
не ломая контракт:

1. **Переписать промт под уровень.** Opus-промт (многошаговый, с reasoning) → для haiku/gemma
   ужать до прямой инструкции + 1–2 few-shot примера. Хранить в роли две версии: `prompt_full` и `prompt_lite`.
2. **Ослабить формат контролируемо.** Если слабая модель не держит строгий JSON-schema — перейти на
   «JSON best-effort + клиентская валидация»; при провале — извлечь ключевые поля регуляркой, остальное
   пометить `null`. Контракт (набор полей) сохраняется, глубина — падает.
3. **Снизить объём работы.** Длинные аналитические задачи → «короткий режим» (как у Пророка:
   `opus` полный / `sonnet` короткий). Меньше шагов reasoning, меньше `max_tokens`.
4. **Честная уверенность.** Если часть советников/источников недоступна — **явно сообщить модели**
   об этом и понизить итоговую уверенность вывода (а не делать вид, что данные полны).
5. **Пометить деградацию.** В ответ добавить служебный флаг `degraded=true, level=<факт.модель>`;
   в лог — `WARNING` (роль, основная→фактическая модель, причина, № попытки).

```python
def build_prompt(role, level):
    # level: "full" для S/A, "lite" для C/local
    return role.prompt_full if level == "full" else role.prompt_lite
```

#### 5.4.3 «Поставить на ноги и позвать админа» (эскалация)

Принцип: **младший брат пытается поднять старшего; если не вышел — зовём владельца.**

```
call_llm(role, messages):
  for model in [primary, *backups, local]:          # лестница из §5.4.1
      for attempt in retry.attempts:                 # transient → backoff 2/4/8
          try:
              r = invoke(model, build_prompt(role, level_of(model)), messages)
              if r.stop_reason == "refusal":          # policy: не ретраить тем же промтом
                  break_to_next_model()
              if model is not primary:                # сработал бэкап
                  log.WARNING(role, primary, model, reason)   # «брат поднял старшего» = WARNING
                  r.degraded = True
              return r
          except Transient(Timeout|429|5xx|Net):
              sleep(backoff); continue                # ещё попытка той же модели
          except Fatal:
              break                                   # эта модель не вывезла → следующая
  # сюда дошли = упали ВСЕ, включая Ollama → каскад
  log.CRITICAL(role, chain, last_error)
  alert_owner(YOUR_TELEGRAM_ID, role, chain, last_error)      # сквозной критический алерт
  return degraded_stub(role)                          # пользователю — явный понятный алерт, НЕ стектрейс
```

**Два уровня сигнала:**

| Ситуация | Лог-уровень | Кому | Текст |
|---|---|---|---|
| Бэкап **справился** (поднял старшего) | **WARNING** | в лог/эфир | `⚠️ <роль>: <основная> недоступна (<причина>), ответ выдан <бэкап>. Деградация уровня.` |
| Бэкап **тоже упал** (каскад) | **CRITICAL** | владельцу + лог | `🔴 <роль>: вся цепочка упала (<opus→sonnet→…→gemma>). Последняя ошибка: <e>. Нужно вмешательство.` |

**Сквозной алерт владельцу** идёт **один раз на инцидент** (дедуп по `role+hour` в Redis, как webhook-дедуп),
чтобы каскад не флудил. Пользователю — `degraded_stub`: «Сервис временно деградировал, специалист уведомлён»,
а не пустой ответ и не трейс.

## 6. Верификация (на отсутствие конфликтов)

| Проверка | Критерий | Статус |
|---|---|---|
| Совместимость промтов бэкапов | у каждой роли есть `prompt_full` и `prompt_lite`; lite не требует фич, которых нет у слабой модели (тяжёлый JSON-schema, длинный reasoning) | ✅ заложено в §5.4.2 |
| Контракт ответа стабилен при деградации | набор полей сохраняется; падает только глубина/формат-строгость | ✅ §5.4.2 п.2 |
| Прозрачность интерфейса fallback | `OllamaResult` имитирует `anthropic` (`.content[0].text`) — бот не различает источник | ✅ уже в `shared/ollama.py` |
| Нет прямых вызовов провайдера | боты ходят через `call_llm()` | ⬜ требует рефактора прямых `messages.create` |
| Дедуп алертов | каскад шлёт владельцу 1 сигнал/инцидент | ✅ §5.4.3 |
| `refusal` не зацикливается | policy-отказ → следующая модель, без ретрая тем же промтом | ✅ §5.4.3 |
| Деньги-adjacent | `tilly-trader` не трогаем кодом, только декларативный конфиг/роадмап | ✅ правило офиса |

**Конфликтов не выявлено.** Главный технический долг для активации стандарта — единая `call_llm()`
вместо точечных `try_ollama → anthropic` (см. §7, статус ⬜).

## 7. Стандарт отказоустойчивости (итоговый, для системы)

1. **Все LLM-вызовы — через `call_llm(role, messages)`.** Прямой `messages.create` в боте запрещён.
2. **Источник истины** — `failover_chains.yaml` (роль → primary/backups/local + retry + escalate_to).
3. **Лестница замен:** primary → бэкап того же класса → downgrade на класс ниже → Ollama → алерт.
4. **Transient** (`Timeout/429/5xx/Net`) — ретрай backoff `2/4/8`; **provider overload 529** — сразу бэкап;
   **policy refusal** — без ретрая, сразу следующая модель.
5. **Деградация = промт под уровень** (`prompt_lite`) + ослабленный формат + честная уверенность +
   флаг `degraded` + **WARNING** в лог.
6. **Каскад** (упали все, вкл. Ollama) → **CRITICAL** + сквозной алерт владельцу (дедуп `role+hour`) +
   `degraded_stub` пользователю (никогда — пустота/трейс).
7. **Модели:** финальные решения — `claude-opus-4-8`; рабочие — `claude-sonnet-4-6` /
   `claude-haiku-4-5-20251001`; локальный рубеж — `gemma3:4b`. Не-Anthropic — точки расширения,
   включаются по ключам, цены сверять перед боем.
8. **Claim субагента о коде/моделях — проверять руками** (`grep`/чтение) до правки (урок офиса).

## 8. Итог для пользователя — кратко

- **Сведена единая лестница мощности** (S→D): `opus-4-8`/`fable-5` (S) → `o3`/`R1`/`sonnet-4-6` (A) →
  `gpt-4o`/`gemini-pro`/`V3` (B) → `haiku-4-5`/`mini`/`flash` (C) → `gemma3:4b` (D).
- **Роли закреплены за моделями:** Стратег/Кодер → opus-4-8; Креатив/Данные → sonnet-4-6;
  Оператор → haiku-4-5; последний рубеж → Ollama.
- **Failover** оформлен как цепочка `primary → бэкап → downgrade → Ollama → алерт` с двумя уровнями
  сигнала: **WARNING** (бэкап поднял старшего) и **CRITICAL** (каскад → владельцу).
- **Graceful degradation:** промт переписывается под уровень модели, формат ослабляется контролируемо,
  уверенность честно понижается — контракт ответа не ломается, бот не падает.
- **Допущения по failover-сценариям зафиксированы** (A1–A5): профиль трафика, SLA, триггеры,
  не-Anthropic цены — ориентир и подлежат проверке.
- **Главный долг для включения:** единая `call_llm()` поверх текущего `try_ollama → anthropic`.

> Не-Anthropic ID/цены — ориентир на дату cutoff (2026-01) и **подлежат сверке с провайдерами**.
> Anthropic-строки и цены выверены по внутреннему справочнику моделей.
