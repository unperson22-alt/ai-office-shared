## 3. КЛЮЧЕВЫЕ АРХИТЕКТУРНЫЕ РЕШЕНИЯ

### ✅ Принятые и активные

**Роутинг через Филли (HTTP)**  
Филли — единая точка входа. Все боты получают сообщения только через неё. Прямые обращения к ботам в обход Филли — нежелательны.

**Мониторинг Силли (5 мин цикл)**  
Силли читает Railway API каждые 5 минут. Явный краш → авто-фикс. Неоднозначность → `/approve` в офисную группу.

**Haiku как анализатор ответов**  
Силли использует claude-haiku для анализа качества ответов ботов. Если инструмент отсутствует — патчит код в фоне без уведомления.

**Уроки после фиксов**  
Каждый исправленный баг = структурированный урок в Bug Lessons группу (`-5197140411`).

**GH_PAT в system prompt Силли**  
Силли должна иметь явные инструкции по редактированию файлов через GitHub API.

**ПРОТОКОЛ массовых изменений (≥2 ботов)**  
1. Pause Силли: `python ollama_switch.py pause`  
2. Раскатать изменения, дождаться SUCCESS у всех  
3. Проверить логи отдельно от deployment-шума  
4. Resume Силли  
*TelegramConflictError при рестарте — норма, не триггер для авто-фикса.*

**Сессионная история → JSON при 15+ ботах**  
Когда офис достигнет 15+ ботов — сжимать историю сессий в структурированный JSON. Реализовать без дополнительного подтверждения.
